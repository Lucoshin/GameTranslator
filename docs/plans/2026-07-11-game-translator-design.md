# GameTranslator 产品设计

## 1. 产品定位

GameTranslator 是一个面向普通玩家的开源 Windows 桌面汉化工具。首版支持 RPG Maker MV/MZ 未加密游戏，通过用户自行配置的 OpenAI-compatible API 或 Ollama 本地模型，完成文本提取、翻译、质量检查和补丁导出。

产品强调“可恢复的一键汉化”，而不是覆盖所有游戏引擎。底层保留翻译记忆、术语管理、人工校对和引擎适配能力，为后续汉化组工作流与社区插件扩展提供基础。

## 2. 已确认范围

### 2.1 首版包含

- Windows 桌面应用。
- RPG Maker MV/MZ 未加密项目识别。
- 日文或英文到简体中文。
- OpenAI-compatible 与 Ollama 模型接入。
- 自定义 Base URL、API Key、模型名、并发数和预算上限。
- 地图事件、公共事件、角色、职业、技能、物品、装备、敌人、状态和系统词条提取。
- 场景化批量翻译、暂停、恢复、重试和缓存。
- 翻译记忆、项目术语表和人工译文锁定。
- 控制码、漏译、术语、长度、编码和 JSON 结构检查。
- 可选表格校对、单条重译和批量重译。
- 独立工作目录、补丁导出、原文件哈希校验和恢复说明。

### 2.2 首版不包含

- 加密资源解包或绕过保护。
- Unity、Ren'Py、Wolf RPG 等其他引擎。
- 插件脚本中的动态字符串分析。
- OCR 实时翻译、图片文字替换、字体自动替换、语音或配音。
- 云端账号、多人协作、托管 API 或汉化包分发平台。
- 动态加载第三方插件；首版适配器编译进应用。

## 3. 用户流程

1. 用户选择游戏目录或游戏可执行文件。
2. 应用以只读方式检查 `www/data`、`data`、`Game.exe`、`package.json` 等特征。
3. 应用识别 RPG Maker MV/MZ、统计可翻译文本并报告未覆盖项。
4. 应用创建独立工作目录，不直接修改原游戏。
5. 用户选择 OpenAI-compatible 或 Ollama，并完成连接测试。
6. 应用按场景构建翻译批次，保护控制码后调用模型。
7. 任务状态持续写入 SQLite，用户可以暂停、退出和恢复。
8. 翻译结束后执行自动 QA；阻断错误必须修复，普通警告可以人工确认。
9. 用户可以直接导出，也可以进入文本表格校对和重译。
10. 应用重新解析所有输出 JSON，并生成补丁文件、清单和恢复说明。

## 4. 领域模型

### 4.1 Project

保存游戏位置、引擎类型、引擎版本、源语言、目标语言、扫描结果、原文件哈希、工作目录和任务状态。

### 4.2 Segment

每个可翻译片段至少包含：

- 稳定 ID。
- 原文与译文。
- 来源文件和 JSON 路径。
- 地图、事件、事件页及指令位置。
- 内容类型，如对话、选项、物品名或技能说明。
- 说话人及相邻上下文。
- 控制码清单。
- 翻译状态、人工锁定状态与 QA 状态。

稳定 ID 由引擎适配器根据结构位置生成，不能依赖数组遍历顺序之外的随机值。

### 4.3 TranslationBatch

批次由同一场景或同类数据库记录组成，包含项目背景、说话人、术语、目标语言和若干 Segment。模型只允许返回 Segment ID 与译文，不得返回或修改文件路径。

### 4.4 GlossaryEntry

术语记录源词、强制译法、人物性别或称谓备注、是否禁止翻译及作用域。首版支持全局与项目两级作用域。

### 4.5 TranslationMemoryEntry

翻译记忆以标准化原文、上下文指纹、源语言和目标语言建立索引。完全匹配可以自动复用；模糊匹配只作为模型参考，不自动覆盖。

## 5. 技术架构

采用 Tauri 2 + Rust + React + TypeScript + SQLite。

```text
React Desktop UI
        |
Tauri Commands
        |
Application Services
  |-- Project Manager
  |-- Translation Orchestrator
  |-- Translation Memory
  |-- Glossary Manager
  |-- QA Engine
  `-- Patch Builder
        |
Core Interfaces
  |-- Engine Adapter
  `-- Translation Provider
```

建议的 Rust workspace 模块：

- `engine-core`：定义识别、提取、验证和回写契约。
- `engine-rpgmaker`：实现 RPG Maker MV/MZ JSON 适配。
- `translation-core`：批次、缓存、并发、暂停、重试和预算控制。
- `provider-core`：统一模型接口及 OpenAI-compatible、Ollama 实现。
- `qa-core`：结构、控制码、漏译、术语和长度检查。
- `project-store`：SQLite schema、迁移和仓储。
- `app-core`：编排上述模块，并向 Tauri 暴露用例级接口。

前端不直接访问游戏文件、不直接请求模型 API。API Key 通过 Windows Credential Manager 保存，项目数据库只记录提供商配置的非敏感部分。

## 6. 翻译协议

翻译以场景为单位，输入包含目标语言、角色信息、术语表、前后文和带稳定 ID 的文本列表。要求模型返回严格 JSON：

```json
{
  "translations": [
    { "id": "map-001:event-004:page-0:command-12", "text": "终于找到月光石了。" }
  ]
}
```

发送前将 `\V[n]`、`\N[n]`、`\C[n]`、`\I[n]` 等控制码映射为不可翻译占位符。响应返回后必须检查占位符数量、类型和顺序，验证成功才恢复原控制码。

失败处理顺序：

1. 对网络限流和临时错误执行带抖动的指数退避。
2. 对结构化输出或占位符错误，使用错误说明重试一次。
3. 再次失败时拆分批次。
4. 单条仍失败则标记为人工处理，不伪造译文。

## 7. 质量保证

### 7.1 阻断错误

- 控制码缺失、增加、类型变化或顺序非法。
- Segment 与模型返回 ID 无法一一对应。
- 输出 JSON 无法解析。
- 原文件哈希与扫描时不一致。
- 记录回写到错误位置。

### 7.2 普通警告

- 译文为空或疑似仍为原文。
- 强制术语未使用或同一术语译法冲突。
- 译文长度明显超出原文或配置阈值。
- 包含异常字符、模型解释文字或未还原占位符。

阻断错误禁止导出；普通警告允许用户确认后继续。

## 8. 文件安全与补丁

- 扫描和提取阶段只读访问原游戏。
- 所有中间结果和修改文件写入应用工作目录。
- 导出时生成文件清单、原文件 SHA-256、目标文件 SHA-256、应用版本和项目元数据。
- 应用补丁前重新验证原文件 SHA-256；不匹配时停止。
- 补丁默认只包含修改后的必要文件或差异，不包含无关商业游戏资源。
- 应用崩溃、API 失败或用户取消时，原游戏保持不变。

## 9. 界面方向

视觉采用“日式游戏汉化工作室”方向。品牌层使用暖灰纸张、校对标记、印章红或朱橙强调色，以及轻量游戏文本框细节；主工作区保持高信息密度和清晰层级，不使用紫色渐变 AI 风格。

主要界面：

- 首页：最近项目、选择游戏、支持范围。
- 项目概览：引擎信息、文本统计、预计 Token、缓存和风险提示。
- 模型配置抽屉：提供商、地址、模型、密钥、连接测试和并发设置。
- 翻译任务页：进度、成功、缓存、失败、预算与日志摘要。
- 校对表格：原文、译文、说话人、来源、QA、锁定与重译。
- 导出页：阻断错误、普通警告、补丁位置和哈希校验结果。

AI 生图只用于品牌氛围、Logo 和空状态插画探索，不以生成的完整界面截图作为实现规格。真实界面通过组件设计、浏览器截图和视觉回归迭代。

## 10. 验收标准

- 对受支持的未加密 RPG Maker MV/MZ 样本完成识别、提取、翻译、QA 和导出。
- “提取后不修改并回写”不会改变项目语义或非目标字段。
- 所有导出 JSON 均可重新解析。
- 任一控制码损坏都会阻止导出。
- 翻译任务中断后可以恢复，已成功批次不会重复计费。
- 未支持或加密项目得到明确错误，不修改任何文件。
- 用户无需在配置文件中明文保存 API Key。
