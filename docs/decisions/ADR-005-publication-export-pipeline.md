# ADR-005: 书籍导出采用统一书稿模型与格式专属出版写入器

## Status
Accepted

## Date
2026-07-14

## Context
书籍译校完成后既需要开放的结构化归档，也需要出版社可编辑稿、电子书和可审阅的印刷定稿。单一 Markdown 文件无法承载出版元数据、章节分页、成品尺寸和电子书导航；直接调用 Office 又会让导出依赖用户机器上的桌面软件。

## Decision
- `BookProject` 保存作者、译者、出版社、ISBN、版权信息和默认成品尺寸，旧项目缺少这些字段时使用向后兼容的默认值。
- `content-book` 分别生成 UTF-8 Markdown、Office Open XML DOCX、EPUB 3 和嵌入中文字体的 PDF，不修改导入源文件。
- PDF 首批支持大32开（140 × 203 mm）、A5（148 × 210 mm）和16开（185 × 260 mm），支持页码与章节另起页。
- DOCX 保留书名、章节和正文样式，正文采用两字首行缩进；EPUB 3 写入元数据、目录、清单和书脊。
- 桌面端只在文件成功写入后追加导出历史。历史记录保存格式、路径、目标语言、时间和版式，可定位文件或按原设置再次导出。
- Windows 发布构建必须启用 Tauri `custom-protocol`，使 EXE 使用内嵌前端资源而不是访问开发服务器。

## Alternatives Considered

### 统一转换为 Markdown 后调用第三方桌面软件
依赖 Office、Pandoc 或浏览器打印环境，难以保证目标机器可用，也无法稳定记录失败边界，因此拒绝。

### 使用一个通用模板同时生成所有格式
DOCX、EPUB 和 PDF 的结构与版式语义不同，强行共享模板会把格式细节泄漏到统一书稿模型，因此采用共享内容模型、独立写入器。

### 把导出历史放在前端状态
应用重启后记录会丢失，也无法作为真实生成文件的审计线索，因此由 Tauri 写入应用本地数据目录。

## Consequences
- 导出链路不依赖用户安装 Office 或电子书工具。
- PDF 会嵌入中文字体，文件体积通常高于纯文本格式，但跨机器显示更稳定。
- 当前输出面向可继续编辑和审稿，不等同于印厂最终制版；出血、专色、PDF/X 和复杂脚注需在后续专业排版阶段处理。
- 新增出版格式时应实现独立写入器，并保留统一历史契约。
