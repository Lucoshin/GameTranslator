# ADR-003: 收口应用服务、适配器注册与可恢复任务边界

## 状态

Accepted

## 日期

2026-07-13

## 背景

MVP 快速扩展到 RPG Maker、Ren'Py 和 RimWorld 后，Tauri 后端同时承担 DTO、Provider 构造、性能策略、任务编排、持久化和系统桥接。功能可用，但业务规则无法被未来 CLI 复用，未完成任务也只存在于进程内存。

## 决策

- `app-core` 是应用策略和稳定错误契约的所有者；Provider 构造、凭据要求和翻译性能策略从 Tauri 下沉。
- `AdapterRegistry` 是支持格式的单一注册入口。首版继续静态编译适配器，不引入动态插件。
- `project-store` 使用 SQLite 保存可恢复任务快照，并显式校验任务状态转换。
- Tauri 根入口只注册模块；Command DTO 和进度事件各自独立，前端共享契约集中在 `src/api`。
- 后端新增能力使用带稳定 `code` 和 `message` 的 `AppError`；前端在迁移期同时接受结构化错误和旧字符串错误。
- 恢复任务时重新扫描原项目，并依靠输入指纹缓存跳过已经成功的 Segment，避免重复调用模型。

## 备选方案

### 重写整个桌面后端

边界最整洁，但会同时改动补丁安装、历史记录和多个引擎路径，回归面过大。未采用。

### 动态插件注册

能在运行时扩展格式，但首版接口和版本策略尚未稳定，会增加安全与发布成本。未采用。

### 只保存前端状态

实现简单，但进程崩溃时无法可靠恢复，且不能保证已完成翻译不重复计费。未采用。

## 后果

- 应用策略可以脱离 Tauri 测试和复用。
- SQLite 成为任务恢复的单一真相来源；系统凭据库仍是密钥的唯一存储位置。
- 已有字符串错误保持兼容，新 Command 应返回结构化错误。
- 旧 `EngineAdapter` API 暂时保留给已有引擎测试和调用者；新内容源统一经 `AdapterRegistry` 进入应用层。
