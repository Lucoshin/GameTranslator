# ADR-001: 采用 Tauri、Rust、React 与 SQLite

## 状态

Accepted

## 日期

2026-07-11

## 背景

GameTranslator 需要安全处理大量本地游戏文件、运行长时间可恢复任务、保存翻译记忆，并提供易用的 Windows 桌面界面。核心逻辑还需要能够被未来的 CLI 和其他引擎适配器复用。

## 决策

采用 Tauri 2 作为桌面容器，Rust 实现核心和文件处理，React + TypeScript 实现界面，SQLite 保存项目、任务、术语和翻译记忆。

前端只能通过用例级 Tauri Command 调用 Rust，不直接操作游戏文件或访问模型服务。Rust 核心按引擎适配、模型提供商、翻译编排、QA 和存储职责拆分。

## 备选方案

### Electron + TypeScript

优点是开发生态成熟、团队容易上手。缺点是桌面包体和内存占用更高，文件解析、并发任务与核心复用容易与 UI 代码混杂。未采用。

### Python + PySide/PyQt

优点是模型生态丰富、原型速度快。缺点是跨平台打包、依赖管理、静态类型约束和长期分发维护成本更高。未采用。

### C# + WinUI/WPF

Windows 集成和凭据管理良好，但会让跨平台和社区前端贡献更困难。由于首版 Windows 优先而非 Windows 独占，未采用。

## 后果

- Rust 增加初期开发门槛，但提高文件安全、性能、类型约束和 CLI 复用能力。
- Tauri 减少桌面包体，同时要求明确维护 Rust 与 WebView 边界。
- SQLite 适合单用户本地工作流；首版不解决多人并发编辑。
- 首版适配器编译进应用，待接口和版本策略稳定后再考虑动态插件。
